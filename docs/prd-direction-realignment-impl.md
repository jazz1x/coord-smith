## 1. Planning Summary

### 1.1 Core Requirements

- v3 기획서의 17개 파일 변경 + 2개 조건부 항목을 **17 commits** (최대) 로 직렬화.
- **Tier sequencing 강제**: P0 → P1 → P2. 각 tier 완료 전 다음 tier 착수 금지.
- 각 커밋은 **one-commit safe**: 사전 점검 통과 → 변경 → 검증 그린 → 커밋. 실패 시 정해진 롤백 절차.
- autoloop 재-쓰기 위험을 최소화하기 위해 **autoloop 가드(C1)가 가장 먼저 랜딩**.

### 1.2 Success Metrics

- 전체 커밋 시퀀스(C1-C17) 완료 후 `uv run pytest -q` = **745 passed, 1 skipped** (기존 744 + 신규 `test_docs_ceiling_truth` 1).
- `ruff check .` / `uv run mypy` 그린.
- v3 기획서 §5 Test Criteria 항목 모두 pass.
- 각 커밋이 **단일 scope**: git show --stat 결과 각 커밋이 §3.1 매핑표의 해당 행만 건드림.
- 어떤 커밋도 rebase/force-push 를 필요로 하지 않음.

### 1.3 Key User Flows

1. **착수 경로**: 운영자/에이전트가 `docs/prd-direction-realignment-impl.md` 를 읽고 C1 부터 순차 실행.
2. **중단-재개 경로**: 임의 커밋 경계에서 중단 → 다음 세션에서 `git log` 확인 후 다음 커밋부터 재개. 각 커밋이 독립적으로 검증된 상태이므로 안전.
3. **롤백 경로**: 특정 커밋 검증 실패 → `git reset --hard HEAD~1` 또는 `git revert <sha>`. 본 PRD 는 `git revert` 선호 (공유 브랜치 안전).
4. **Tier 경계 검증**: P0 종료 시 mid-run 검증, P1 종료 시 mid-run 검증 (§5.2).

### 1.4 Non-Functional Requirements

- **Atomicity**: 각 커밋이 그린 상태만 남김. 실패 시 stash 또는 롤백.
- **Traceability**: 커밋 메시지에 해당 C-번호와 v3 매핑 행 번호 포함 (`C3 / v3 §3.1#2`).
- **No hooks bypass**: `--no-verify` 금지. pre-commit 실패는 근본 수정.
- **Concurrent safety**: autoloop 가드가 C1 로 선행 랜딩하므로 C2-C17 실행 중 자율 루프가 `work-rag.json` 을 덮어쓰지 않음.

## 2. Technical Design

### 2.1 Scope

- v3 기획서 §3.1 의 19개 행 중 `#14 (ledger preserve)` 과 `#18 (verify-only)` 는 커밋을 생성하지 않음. 실행 커밋 = **17**.
- 커밋 체인은 단일 브랜치 `codex/python-311-baseline` 위에 쌓는다 (현 작업 브랜치).
- 브랜치 분리 고려: P0 만 먼저 PR 분리 가능. 본 PRD 는 단일 브랜치를 가정하되 `§6 Guardrails` 의 soft rule 로 PR 분리 옵션 명시.

### 2.2 Pre-flight Verification (공통)

모든 커밋 공통 사전 점검:

```bash
git status --porcelain           # working tree clean 확인
git branch --show-current        # codex/python-311-baseline
uv run pytest -q                 # 베이스라인 그린 확인
```

working tree 가 dirty 면 먼저 정리(stash/commit) 후 착수.

### 2.3 Rollback Philosophy

- 커밋 **직후** 검증 실패 → `git reset --hard HEAD~1` (로컬에 한정).
- 커밋 **push 후** 검증 실패 → `git revert <sha>`.
- `rag.json` / `work-rag.json` 개정 실패 → 반드시 `git revert` (로컬 reset 가능하지만 history 일관성 유지 위해 revert 권장).
- 연쇄 실패 → tier 경계까지 순차 revert. Tier 경계 미만의 부분 revert 는 허용 (각 커밋이 독립 scope 이므로).

## 4. Edge Cases & Error Handling

### 4.1 정상 경로

| 시나리오 | 기대 | 처리 |
|---------|------|------|
| C1..C17 순차 랜딩 | 각 커밋 그린 | 템플릿대로 |
| 커밋 경계 중단 후 재개 | 다음 세션에서 재개 안전 | §1.3 flow 2 |

### 4.2 경계 / 오류

| 시나리오 | 위험 | 대응 |
|---------|------|------|
| C1 가드 이미 존재 | 중복 코드 | C1 스킵(verify-only) 후 C2 로 |
| C2 atomic 실패 (rag.json 만 편집, archive 누락 상태 커밋) | durable history 훼손 | 즉시 `git revert`, 재시도 |

## 6. Guardrails

### Soft (project)

- 커밋 분할 권장: P0 를 먼저 독립 PR 로 보내면 리뷰 부담 감소. 단일 브랜치로 진행해도 허용.
- 커밋 간격에서 autoloop 실행 금지 (C1 이전 구간 특히). C1 랜딩 후에는 허용.
- 커밋 메시지는 `C#` + `v3 §3.1#<row>` 를 반드시 포함 — 추적성 확보.
- P1 의 C10/C11/C12 는 매치가 없으면 **no-op 스킵** (불필요한 커밋 생성 금지).

### Hard (prohibitions)

- `--no-verify` / `--no-gpg-sign` 금지.
- `git push --force` 금지 (main/master 및 공유 브랜치).
- Tier 순서 역전 금지: P1/P2 를 P0 완료 전에 커밋 금지.
- v3 기획서의 Hard prohibitions 전승: ceiling 축소, LLM inference in runtime, Playwright/CDP/Chromium backend, 제3 캐논 메모리, FINAL_STOP 사유 없는 덮어쓰기, legacy TypeScript 부활.
- rag-archive append 없이 rag.json lesson 편집 금지 (§3.3 atomic 위반).
