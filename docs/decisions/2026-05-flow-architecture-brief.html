Architectural Decision Brief

1회 invocation = 1 click 제약을 풀어야 한다.

사용자가 짚은 진짜 사용성 3축 — 시간 정합성 · 좌표/이미지 fallback · flow chaining — 중에서 처음 둘은 코드에 거의 들어 있고, flow가 미해결이다. 진입축은 여기 하나에 모인다.

Verdict

Recipe를 multi-step DSL로 확장하고, 단일-step 호출은 stateless escape hatch로 보존. Stateful session(stdin/socket)은 거부 — 직전에 정리한 단일 진입의 단순성을 다시 무너뜨린다.

01현재 — 1회 invocation = 1 click

호출자가 recipe를 짜고, coord-smith가 12-mission lifecycle 한 번을 돌린다.

external head — OpenClaw / 다른 LLM 에이전트 / 손작성
↓recipe.yaml 작성·전달
coord-smith CLI — --click-recipe path · --session-ref · --target-page-url
↓
12-mission LangGraph · attach → prepare → benchmark → ready → sync → actionable → armed → trigger → click_dispatch → completion → success → done
↓
artifacts/runs/<id>/ · action-log JSONL · screenshot PNG · transition diff
↓exit code 0/1/2/3

12개 mission은 한 번의 click의 lifecycle이다. 여러 click의 sequence가 아니다.

02사용성 3축 — 어디까지 풀려있는가

A
시간 정합성 (Timing coherence)
click 전후 settle, page-transition pixel diff, post-click signal polling. Recipe에서 선언적으로 켜면 된다.
대체로 충족
갭 — settle 시간 하드코딩 (configurable 아님)
B
좌표 + 이미지 fallback
payload > recipe coord > recipe image > no-click 우선순위는 있다. 하지만 한 step 안에서 "image 매칭 실패 시 coord로 fallback" 같은 try-then-fallback chain은 없다.
선언만 됨
갭 — step-내 retry-fallback DSL 부재
C
Flow chaining (coord1 → coord2 → coord3)
현재 1 invocation = 1 click. 여러 클릭을 하나의 흐름으로 표현할 채널이 없다 — 외부에서 N번 호출하거나, 아키텍처를 바꿔야 한다.
미해결
진짜 fork는 여기

03Axis C — 무엇을 풀어야 하는가

"좌표1에서 좌표2 인식을 시키는 건 어떻게 할 것인가?"

코드 사실: 1번 클릭 후 페이지가 바뀌면, 다음 anchor는 화면 위에 새로 나타난 image template으로 잡는 게 자연스럽다 — DOM이 안 열리는 환경(anti-bot, 데스크톱 앱)에서도 일관됨. 즉 답은 image-template anchoring per step. 이걸 declarative로 표현하려면 recipe가 step 시퀀스가 되어야 한다.

043 Options — 비교

Option 1

Multi-step recipe
한 invocation에서 N steps 직렬

Recipe가 step list. coord-smith가 step 사이의 settle/wait/match를 직접 처리. 1 recipe = 1 flow.

  • 결정적 — 외부 추론 라운드트립 0
  • 한 번 작성, 끝까지 자동
  • artifact가 한 트리에 묶임
  • mid-flow 분기 표현 제한
  • schema 확장 필요
Option 2

Stateless chain
현재 구조 유지, head가 매번 호출

Step마다 head가 직전 screenshot 보고 다음 recipe 작성. 1 click = 1 invocation.

  • head 추론력 최대 — 분기 자유
  • 현재 invariant 0건 깨짐
  • Option 1과 공존 가능 (1-step recipe)
  • per-step subprocess 비용
  • 브라우저 세션 owner 모호
Option 3

Stateful session
long-running coord-smith

stdin / socket으로 click-at-a-time. 세션 lifetime이 클라이언트와 묶임.

  • Option 1+2의 장점 융합
  • per-step 오버헤드 없음
  • 단순성 후퇴 — 직전에 정리한 MCP 스캐폴드의 재발
  • "한 번 실행, exit cleanly" invariant 깨짐
  • 동시 실행/격리 새 문제 도입

05권장 = Option 1 + Option 2 (escape)

Multi-step recipe가 1차 시민. 단일-step 호출은 그대로 작동 — 0 deprecation.

이유 셋:

  1. 실제 사용 패턴 대부분이 결정적 시퀀스다. 티켓팅 "buy → seat → confirm → pay" 같은 흐름은 head가 step마다 끼어들 일이 적다. Option 1로 충분.
  2. Option 2는 Option 1의 부분집합으로 자연스럽게 표현된다. 단일-step recipe N번 호출 = stateless chain. 즉 head가 끼어들고 싶을 때 그대로 쓸 수 있다.
  3. Option 3은 단순성 후퇴. 방금 MCP 스캐폴드를 영구 out-of-scope로 못박은 그 결정과 같은 결의 결정을 다시 깨는 셈. spec 일관성이 중요하다.

예상 recipe 모양

version: 1 steps: - name: open-buy image: templates/buy-button.png after: wait_for: { image: templates/seat-panel.png, timeout: 5 } - name: select-seat image: templates/available-seat.png region: [200, 300, 800, 600] fallback: # axis B 통합 coord: { x: 400, y: 450 } - name: confirm image: templates/confirm.png verify_transition: true

06다음 작업 — Spine PRD 미리보기

PRD: multi-step flow recipe + step-level fallback
draft · 1 PR
scope
  • recipe schema 확장: steps: list[Step] (기존 missions: dict와 공존)
  • Step DSL: name · image | coord · region · confidence · fallback · post_click_signal · verify_transition · retry?
  • adapter: step list 직렬 실행, 사이 settle/wait, fallback chain 시도, 모두 실패 시 typed error + 부분 artifact
  • 12-mission graph 안의 click_dispatch 자리에 "step list 실행" 노드로 surface 변경
  • 단일-step recipe 그대로 동작 (backwards-compat 검증)
  • tests: 단일 step 회귀 / 3-step happy / mid-step fallback / mid-step 실패 + 부분 artifact
out of scope
  • stateful session (Option 3 — 영구 거부)
  • mid-flow head reasoning (head가 끼어들고 싶으면 step 끊어 호출)
  • DPI/resolution 정규화 — 별 PRD
  • retry/jitter — 별 PRD (anti-bot 대비 시)
success
  • 3-step recipe로 tickets.interpark.com 같은 anti-bot 사이트에서 buy → seat → confirm 한 번에 자동 실행
  • 코드 invariant 0건 깨지지 않음
  • 단일-step 사용자 누구도 마이그레이션 필요 없음

Confirm — 의사결정자 입력 필요

  • Option 1 + Option 2 escape 노선에 동의? 동의 시 Option 3은 영구 out-of-scope로 prd.md §Non-Goals에 명시.
  • Step DSL에서 image-first 강제 (coord는 fallback)? 현재는 image와 coord가 동등 우선순위. step 단위로 좁히면 portability ↑, 단일-step 사용자에게 친숙도 ↓.
  • Multi-step을 12-mission graph 안에 끼울지 / 12-mission을 step list 위 1 step으로 격하? 전자 = 변경 작음, surface 추함. 후자 = 깔끔, mission contract 재정의 필요.
