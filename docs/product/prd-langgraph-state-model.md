# PRD — LangGraph State Model

## Purpose

This PRD defines the canonical `LangGraph` state model for the Python-first
`ez-ax` runtime.

It exists so an autonomous agent can determine:

- what runtime state exists
- which state is global versus mission-local
- what each graph node reads and writes
- how transitions are allowed to progress
- when the graph retries, stops, or escalates
- how released and modeled paths remain separated

This PRD should be read beneath:

1. [`prd-e2e-orchestration.md`](./prd-e2e-orchestration.md)
2. [`prd-python-langgraph-runtime.md`](./prd-python-langgraph-runtime.md)
3. [`prd-runtime-missions.md`](./prd-runtime-missions.md)

Precedence rule:

- if a graph transition in this PRD appears to widen the released ceiling, the
  benchmark PRD wins
- if a graph behavior in this PRD conflicts with the mission map, the mission
  map wins
- if a graph behavior in this PRD conflicts with the ownership model, the
  Python runtime PRD wins

## State Model Goals

The graph state model must make the runtime:

- deterministic
- resumable from artifacts and RAG
- explicit about released versus modeled progression
- explicit about retry versus stop behavior
- explicit about what evidence is still missing

State-model rule:

- no node may depend on hidden operator intuition or implicit memory

## State Layers

The runtime state is divided into:

- global run state
- mission-local state
- evidence state
- control state
- memory state

Layer rule:

- global state holds durable run truth
- mission-local state holds the smallest temporary state needed for the current
  mission
- evidence state holds normalized comparable evidence
- control state holds retry, stop, and release-gate decisions
- memory state holds RAG update inputs and compression boundaries

## Global Run State

Global run state must include:

- `run_id`
- `run_status`
- `current_phase`
- `current_mission`
- `current_anchor`
- `highest_reached_stage`
- `approved_scope_ceiling`
- `release_status`
- `session_ref`
- `site_identity`
- `target_page`
- `final_artifact_bundle_ref`

Global-state rule:

- `approved_scope_ceiling` remains `pageReadyObserved` until the benchmark PRD
  explicitly changes
- `release_status` must distinguish `released`, `modeled`, and `control-only`

### Run Root Ownership (Released Scope)

In released scope (up to `pageReadyObserved`), the orchestrator/graph must own a
single run-root directory that contains the run's artifacts.

Contract:

- `run_id` is generated by `ez-ax` at the released-scope graph entrypoint
  (not by `OpenClaw`)
- the graph entrypoint MUST create and own `run_root` under a stable base path
  (relative to repo root):
  - `artifacts/runs/{run_id}/`
- `final_artifact_bundle_ref` MUST reference this run root (filesystem path
  string) so validators can resolve evidence refs to concrete artifacts without
  operator guesswork

Scope rule:

- run-root ownership and artifact creation must not imply any modeled
  post-ceiling workflow stage occurred; it is only the container for released
  ceiling artifacts and deterministic stop evidence

## Mission-Local State

Mission-local state may include:

- current mission inputs
- current mission outputs
- current mission checkpoint
- current mission evidence envelope
- mission attempt count
- current mission stop reason candidate

Mission-local rule:

- mission-local state must be reset or archived when the graph leaves the
  mission
- no future node may rely on mission-local state that was not promoted into
  global or evidence state

## Evidence State

Evidence state must include normalized artifacts for:

- DOM or text signals
- clock or timing signals
- action logs
- screenshot or vision fallback artifacts
- coordinate-action records when used
- checkpoint timestamps
- artifact comparison references

Evidence-state rule:

- typed primary signals remain the first-class truth source
- screenshot evidence may support or unblock interpretation, but it may not
  replace typed primary signals by itself
- coordinate evidence documents execution, not state truth, unless another
  typed source confirms the state

## Control State

Control state must include:

- current stop reason
- retry policy snapshot
- retry counter
- escalation flag
- release-gate decision
- validation status
- skipped-validation note when applicable

Control-state rule:

- stop reasons must be typed and comparable
- retry state must never be inferred from a missing artifact

## Memory State

Memory state must include:

- `work_rag_update_pending`
- `lesson_promotion_pending`
- `compression_boundary_pending`
- latest evidence references for RAG writing

Memory-state rule:

- the graph must be able to close a task without consulting hidden thread
  memory

## Node Contract

Every graph node must declare:

- node name
- owned mission
- required read state
- guaranteed write state
- required evidence inputs
- produced checkpoint or control output
- success transition
- stop transition
- retry transition if allowed

Node-contract rule:

- a node may only write state that its mission or control role owns
- browser-facing side effects must be mediated through OpenClaw-owned execution
  requests

## Mission Node Mapping

Released mission nodes:

- `attach_session_node`
- `prepare_session_node`
- `benchmark_validation_node`
- `page_ready_observation_node`

Modeled mission nodes:

- `sync_observation_node`
- `target_actionability_observation_node`
- `armed_state_entry_node`
- `trigger_wait_node`
- `click_dispatch_node`
- `click_completion_node`
- `success_observation_node`
- `run_completion_node`

Control nodes:

- `release_gate_evaluation_node`
- `retry_or_stop_decision_node`
- `work_rag_update_node`
- `work_rag_compression_node`
- `lesson_promotion_node`
- `e2e_replay_or_comparison_node`
- `python_validation_execution_node`

Node-mapping rule:

- mission names from [`prd-runtime-missions.md`](./prd-runtime-missions.md)
  remain the canonical identifiers
- node names exist to express graph structure, not to rename missions

## Transition Model

The graph supports three transition classes:

- forward mission progression
- control wrapping or closure
- retry or stop branching

## Released-Ceiling Transition Slice

This section isolates the approved transition slice so the current released
ceiling can be read without scanning modeled stages.

Released-ceiling scope:

- the only released forward path is the ordered sequence from
  `attach_session_node` through `page_ready_observation_node`
- any transition that would require `sync_observation_node` or later is modeled
  and must not be executed under the current approved ceiling

Released-ceiling reading rule:

- if a reader needs to validate only the approved ceiling, they may stop after
  the released-path ordering list and treat everything below as modeled-only

Forward-progression rule:

- released-path transitions must be linear through `page_ready_observation_node`
- modeled-path transitions may only begin after the released ceiling has been
  reached or intentionally stopped at
- control nodes may observe or close missions, but they must not silently skip
  required mission transitions

Released-path ordering:

1. `attach_session_node`
2. `prepare_session_node`
3. `benchmark_validation_node`
4. `page_ready_observation_node`

Modeled-path ordering:

1. `sync_observation_node`
2. `target_actionability_observation_node`
3. `armed_state_entry_node`
4. `trigger_wait_node`
5. `click_dispatch_node`
6. `click_completion_node`
7. `success_observation_node`
8. `run_completion_node`

## Edge Preconditions

Every forward edge must validate:

- required predecessor checkpoint exists
- required evidence is present
- released versus modeled status remains consistent
- current stop reason is empty
- no unresolved retry decision is pending

Precondition rule:

- missing evidence is not equivalent to negative evidence
- an edge may not advance only because time passed

## Retry Model

Retries are allowed only when:

- retry policy explicitly permits the mission to retry
- the failure is typed and attributable
- the next attempt can produce meaningfully different evidence

Retries are not allowed when:

- the failure is actually a scope boundary
- the graph would be repeating the same action without new evidence
- released versus modeled status would be blurred by retrying

Retry-model rule:

- retries must increment mission attempt count and preserve prior evidence

## Stop Model

The graph must stop when:

- the released ceiling has been honestly reached
- a typed stop reason exists and retry is not allowed
- policy is insufficient to continue honestly
- evidence remains ambiguous after allowed fallback paths

Stop-model rule:

- the graph stops with a typed stop reason, not a generic failure string

## Escalation Model

The graph escalates when:

- a policy decision cannot be made from existing PRDs
- conflicting evidence cannot be resolved by documented fallback rules
- continuing would require widening scope or changing release claims

Escalation-model rule:

- escalation is distinct from retry and distinct from terminal stop

## Release-Gate Placement

`release_gate_evaluation_node` may run:

- after the released path completes
- after a modeled path produces a comparable artifact bundle

Release-gate rule:

- release-gate evaluation may classify outputs, but it may not relabel modeled
  missions as released

## RAG Integration

The graph must explicitly route task closure through:

1. `work_rag_update_node`
2. `lesson_promotion_node` when warranted
3. `python_validation_execution_node`
4. task-level completion or stop closure

At milestone or phase boundaries, the graph must additionally route through:

- `work_rag_compression_node`

RAG-integration rule:

- a task is not closed until its current-tense work memory is updated and the
  required validation outcome is recorded

## Resumption Model

The runtime must be resumable from:

- current global run state
- retained evidence state
- retained control state
- `work-rag.json`
- `rag.json` when durable lessons are relevant

Resumption rule:

- if resumption would require hidden conversational context, the state model is
  incomplete

## Completeness

This state model is complete only when:

- every mission has a corresponding node identity
- state ownership is explicit
- transition rules are explicit
- retry, stop, and escalation rules are explicit
- released and modeled progression remain distinguishable
- later layout and implementation PRDs can use this model without inventing
  graph semantics
